# Pre-grill — PRD 可行性与 Goal 启动准备

## 概述

全自动交付的**启动前**人机循环。在用户提交 PRD（及可选决定性文件）后，Agent 做项目分析与可行性判断，输出准备清单，**仅在可行且用户确认后**加载 `goal-document` 生成 Goal 文档，并给出 `/goal` 启动指令。

**主产出：Goal 文档**（能驱使 `/goal` 无人值守不中断）。准备清单与可行性结论是中间产物，不是终点。

**硬规则：**

1. **不改 PRD 目标** — 只做可行性分析、缝隙补全建议、准备清单；不重写/降级/删除 PRD 目标
2. **不写产品代码** — 本 skill 只读材料、写分析/清单/触发 Goal 文档生成
3. **不跳过用户确认直接 `/goal`** — Agent 只准备；`/goal` 由用户自己触发
4. **不可行则禁止启动 `/goal`** — 输出原因 + 修正建议，等用户改 PRD，循环直到可行

## 何时使用

- 用户发消息 + PRD 文件路径/内容，准备进入全自动交付
- 用户明确要求「开跑前 grill / 可行性 / 准备 Goal」
- 从零启动一条 Superpowers 全自动链路（pre-grill → Goal → `/goal`）

**不适用：**

- 已在 `/goal` 运行期（门禁 FAIL 走 `auto-recovery`；PRD 矛盾走 `decision-arbiter`）
- 仅改 wiki / 仅做代码实现、无新 PRD 开跑
- 用户已有确认过的 Goal 文档且只想重跑（可跳过本 skill，直接 `/goal`）

## 输入契约

| 输入 | 必需 | 说明 |
|------|------|------|
| PRD 路径或正文 | 是 | 用户产品需求；目标最高优先级 |
| 决定性文件 | 否 | 规格/约束/接口/设计片段等 |
| 项目根 | 否 | 默认为当前工作区根 |
| 既有 wiki | 否 | 存在则必读（见步骤 2） |

## 硬顺序流程（禁止跳步）

必须严格按下列顺序执行。**任一步未完成不得进入下一步**；尤其禁止在用户确认前输出可执行的 `/goal` 启动指令并暗示可以立刻开跑。

```text
1. 读用户 PRD + 附带决定性文件
2. 若存在 wiki：读 README、L1、P1、L6（框架任务再读 P3）
3. 可行性分析：
   - 目标是否可测
   - 是否有致命矛盾
   - 是否缺环境/权限/数据
4. 不可行：
   - 输出原因 + 修正建议
   - 等待用户改 PRD
   - 禁止启动 /goal
   - 循环直到可行
5. 可行：
   - 输出准备清单（权限/配置/环境/goals.max_turns/YOLO 等）
   - 等待用户确认清单完成
6. 加载 goal-document skill 生成 Goal 文档
7. 请用户确认 Goal 文档
8. 输出启动指令：
   /goal 根据 Goal 文档 {path} 无人值守完成：{声明}
```

### 步骤 1：读 PRD 与决定性文件

1. 定位 PRD（路径优先；用户粘贴正文则先落盘或明确引用路径）。
2. 提取并列表：
   - **目标清单**（可测表述优先）
   - **非目标 / 边界**
   - **验收线索**（场景、命令、产物）
   - **环境/依赖暗示**
3. 读取用户附带的决定性文件；记录路径列表，供 Goal 元信息使用。
4. **禁止**在本步改写 PRD 目标。

**完成标准：** 能复述 PRD 全部目标与非目标；决定性文件路径列表就绪。

### 步骤 2：读 wiki（若存在）

按路径探测；**存在则读，不存在则记录「无 wiki」并继续**：

| 文档 | 典型路径 | 用途 |
|------|----------|------|
| README | `README.md` 或 wiki 索引 | 项目定位、启动方式 |
| L1 | `docs/wiki/L1-*.md` 或项目约定路径 | 架构/总览 |
| P1 | `docs/wiki/P1-产品决策日志.md` 等 | 历史产品决策 |
| L6 | `docs/wiki/L6-*.md` 等 | 运维/环境/部署约束 |
| P3（仅框架任务） | `docs/wiki/P3-*.md` | 框架演进约束 |

读取时标注与 PRD **一致 / 互补 / 冲突** 的要点。冲突记入可行性分析，**不得擅自用 wiki 覆盖 PRD 目标**。

**完成标准：** 已读存在的 README/L1/P1/L6（及必要时 P3），并有冲突/约束摘要。

### 步骤 3：可行性分析

从三个维度判定 **可行 / 不可行**。任一维度致命失败 → **不可行**。

#### 维度 A：目标可测

| 检查 | 通过信号 | 失败信号 |
|------|----------|----------|
| 目标可观察 | 每项目标有可验证结果（接口、UI、命令输出、文件产物） | 纯口号、无法判定 done |
| 验收可落地 | 能写出至少 1 条验收场景/目标（可后补细节） | 无法想象证据形态 |
| 范围有界 | 非目标或边界能区分 in/out | 无限范围、无法收束 |

#### 维度 B：无致命矛盾

| 检查 | 通过信号 | 失败信号 |
|------|----------|----------|
| PRD 内部 | 目标之间不互斥 | 同时要求 A 与非 A |
| PRD vs 非目标 | 目标不落在非目标内 | 目标与非目标直接冲突 |
| PRD vs wiki/P1 | 无硬约束冲突，或冲突可在不改目标下细化执行 | 必须改 PRD 目标才能和解 |

#### 维度 C：环境 / 权限 / 数据

| 检查 | 通过信号 | 失败信号 |
|------|----------|----------|
| 环境 | 所需运行时/服务可获得或清单可补齐 | 依赖不可得且无替代 |
| 权限 | 读仓、测、写 `docs/superpowers/` 等权限可满足 | 无权限且无法由用户补齐 |
| 数据 | 样例/密钥/账号可准备或明确由用户提供 | 缺关键数据且不可构造 |

**输出格式（可行性结论）：**

```markdown
## 可行性结论

- **判定：** 可行 | 不可行
- **目标可测：** PASS | FAIL — {摘要}
- **致命矛盾：** 无 | 有 — {摘要}
- **环境/权限/数据：** 就绪 | 缺 — {摘要}
- **阻断项：** {无 | 列表}
- **非阻断缝隙：** {可在准备清单或 Goal 中补全的项}
```

**完成标准：** 三维度均有明确 PASS/FAIL 与摘要；判定二选一。

### 步骤 4：不可行分支（禁止 `/goal`）

若判定 **不可行**：

1. 输出阻断原因（对应维度 A/B/C）。
2. 给出**具体修正建议**（改哪类表述、补什么验收、消什么矛盾、要什么环境）——建议针对 PRD/环境，**不替用户改 PRD 文件目标**。
3. **明确声明：禁止启动 `/goal`。**
4. **停止推进**；等待用户修改 PRD（或补材料）后重新从步骤 1 循环。
5. 不得加载 `goal-document`，不得输出「可以开跑」的启动指令。

**完成标准：** 用户已看到原因 + 建议；流程停在等待，而非 Goal 生成。

### 步骤 5：可行 → 准备清单 + 用户确认

若判定 **可行**：

1. 输出准备清单（可按任务裁剪，但下列项应显式覆盖或标 N/A）：

```markdown
## 准备清单

- [ ] PRD 路径可读且目标可测（本轮已判定可行）
- [ ] 所需环境/依赖/端口/服务已就绪
- [ ] 所需密钥/账号/样例数据已就绪（或确认不需要）
- [ ] 权限足够：读仓库、跑测试、写 docs/superpowers/
- [ ] Hermes Goal 模式可用；建议 `goals.max_turns` ≥ 100
- [ ] 是否启用 YOLO / 自动批准工具已明确
- [ ] 相关 wiki（若存在）可读：README、L1、P1、L6
- [ ] 决定性文件路径已确认
- [ ] 其他任务特有项：{...}
```

2. **等待用户确认清单已完成**（或用户声明「清单完成 / 可以生成 Goal」）。
3. 用户未确认前：**不得**进入步骤 6。

**完成标准：** 用户明确确认准备清单完成。

### 步骤 6：加载 goal-document 生成 Goal 文档

1. 加载 `goal-document` skill。
2. 传入：
   - PRD 路径
   - 决定性文件列表
   - wiki/P1 要点（若有）
   - 本轮可行性结论
   - 已确认的准备清单
3. 按 `goal-document` 产出：
   - `docs/superpowers/goals/YYYY-MM-DD-<topic>-goal.md`
   - 一句话 Goal 声明
   - 完成契约骨架（Goal 内嵌 + `contracts/` 副本，以 goal-document 约定为准）
4. 本 skill **不手写替代** Goal 全文逻辑；以 `goal-document` 为权威填充流程。
5. 若 `goal-document` 尚未安装：按 `templates/goal-document.md` 手工等价填充，并在输出中注明「goal-document skill 缺失，已用模板回退」。

**完成标准：** Goal 文档路径存在且可读；含目标清单、阶段流程、门禁/恢复引用、准备清单、Goal 声明。

### 步骤 7：用户确认 Goal 文档

1. 向用户展示 Goal 文档路径 + Goal 声明 + 关键边界摘要。
2. 请用户确认：目标覆盖正确、边界可接受、路径正确。
3. 用户要求修改时：
   - **不改 PRD 目标**
   - 可调整 Goal 文档中的执行细化、场景骨架、准备清单表述
   - 重大目标变更 → 回到步骤 1（视为新 PRD 轮次）
4. 用户未确认前：**不得**输出最终启动指令（步骤 8）。

**完成标准：** 用户明确确认 Goal 文档。

### 步骤 8：输出 `/goal` 启动指令

仅当步骤 1–7 全部满足后输出：

```text
/goal 根据 Goal 文档 {path} 无人值守完成：{一句话声明}
```

同时提醒：

- 由**用户**在 Hermes 中执行上述 `/goal`；Agent 不代替用户开跑
- 执行 Agent 第一步必须全文阅读 Goal 文档
- 运行期门禁走 `gatekeeper` + `delegate_task`；失败走 `auto-recovery`
- 建议 `goals.max_turns` ≥ 100

**完成标准：** 启动指令已给出；本 skill 职责结束。

## 边界（再次强调）

| 允许 | 禁止 |
|------|------|
| 读 PRD / wiki / 决定性文件 | 改 PRD 目标、降级验收意图 |
| 可行性分析 + 修正建议 | 不可行时启动 `/goal` |
| 准备清单 | 跳过用户确认直接 `/goal` |
| 触发 `goal-document` 写 Goal | 写产品业务代码 |
| 补全执行缝隙（路径、清单、场景骨架建议） | 用 wiki 覆盖 PRD 目标 |
| 等用户改 PRD 后重跑本流程 | 假装环境/权限已就绪 |

## 与其它 skill 的关系

| Skill | 关系 |
|-------|------|
| `goal-document` | 可行且清单确认后由本 skill 触发；产出 Goal 文档 |
| `completion-contract` | Goal 内嵌/复制契约骨架；Intake 后正式跟踪 |
| `gatekeeper` | 运行期阶段门；本 skill 不替代 Intake 门禁 |
| `auto-recovery` | 运行期失败恢复；PRD 不可行不走它 |
| `decision-arbiter` | 运行期 PRD 矛盾/缺口；pre-grill 阶段矛盾应挡在开跑前 |
| `wiki-checkpoint` | 运行期 wiki 合规；pre-grill 只读 wiki |

## 输出模板（可行路径摘要）

每轮与用户交互建议使用固定块，便于扫读：

```markdown
=== Pre-grill 状态 ===
阶段：读材料 | 可行性 | 等改 PRD | 准备清单 | 生成 Goal | 等确认 Goal | 可启动
判定：可行 | 不可行 | 未判定
PRD：{path}
Wiki：已读 README/L1/P1/L6 | 无 wiki
Goal：{path 或 尚未生成}
下一步：{明确等待用户的动作}
```

## Common Pitfalls

1. **跳过 wiki 就判可行** — 存在 L6/P1 时可能隐藏环境/决策硬约束。
2. **把「建议」写成改 PRD** — 不可行时只给建议，由用户改；Agent 不直接改目标。
3. **清单未确认就生成 Goal / 输出 `/goal`** — 违反硬顺序；环境缺失应在开跑前暴露。
4. **把 pre-grill 当实现阶段** — 禁止写产品代码、禁止提前进 Spec/实现。
5. **不可行仍「先跑起来再说」** — 明确禁止 `/goal`，循环直到可行。
6. **Goal 确认前把启动指令当已授权** — 步骤 8 仅在用户确认 Goal 之后。
