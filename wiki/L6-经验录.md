# L6-经验录 — 踩过什么坑

> 所有 Agent 的"避坑指南"。记录设计决策、已知问题、调试技巧、教训。

## 关键设计决策

| 决策 | 结论 | 原因 |
|------|------|------|
| GitHub 是总仓库，正式站点是展示面 | 所有 raw/data/reports/site data 回写 GitHub | 双重保险：GitHub 保留完整历史，站点展示当前结果 |
| 全自动评分，取消人工审核 | curated 不再代表人工审核 | 维护者不再逐条审核，只做方向调整 |
| 来源透明与质量分级 | source_quality=verified/fallback/unverified/blocked | 用户知道数据来源的可信度 |
| Official 与 Ecosystem 分离 | ranking_scope 区分 official/ecosystem | 官方工具不应该与第三方插件混排 |
| 部署职责分离 | Hermes cron 负责生产部署，Actions 只验证 | 避免 Actions 环境暴露服务器路径 |
| 可复用性设计 | 脚本通用，配置在 data/ 和 config/ | 替换 4 个文件即可追踪其他生态 |
| 安全优先 | HSTS/CSP/Content-Security-Policy 严格模式 | 公开站点必须防范 XSS 和注入 |
| 前端零依赖 | 纯原生 HTML+JS+CSS，无框架 | 安全性高、部署简单、无 npm 供应链风险 |

## 已知坑

### 1. Exa 依赖 mcporter
- **现象**：mcporter 或 Exa 服务不可用 → 自动 fallback 到 web 搜索 → 质量显著降低
- **原因**：Exa 是唯一语义搜索源，web 搜索是通用搜索引擎
- **解决方案**：mcporter 配置 exa MCP server，定期检查 exa-status-and-fallback.md 报告
- **发现日期**：项目初始化时

### 2. 分类过宽
- **现象**：agent-harness（363 条）和 testing-review-ci（357 条）分类偏高
- **原因**：分类规则关键词太宽泛，例如 "multi-agent" 匹配过多项目
- **解决方案**：调优 normalize.py 中的 CATEGORY_RULES，增加排除规则
- **发现日期**：2026-07-06

### 3. 双语不完整
- **现象**：源数据 data/projects.yaml 尚未持久化 i18n 字段
- **原因**：i18n 只在 build_site.py 中防御性补齐，没有回写入 projects.yaml
- **解决方案**：在 finalize_data.py 中回写 i18n 字段到 projects.yaml
- **发现日期**：2026-07-06

### 4. raw 数据膨胀
- **现象**：每天保存原始 JSON 快照到 Git 仓库
- **原因**：保留来源证据的需求 vs Git 仓库大小
- **解决方案**：已有 archive_raw.py 脚本，定期归档旧数据
- **发现日期**：2026-07-06

### 5. fallback 噪声
- **现象**：fallback-web 结果可能包含完全无关内容
- **原因**：web 搜索没有领域过滤
- **解决方案**：auto-rejected 规则会过滤低分记录，但仍有噪声残留风险
- **发现日期**：2026-07-01

### 6. GitHub Actions 无 Exa
- **现象**：Actions 环境没有 mcporter/exa token → exa-limit 参数无效
- **原因**：mcporter 需要 agent-reach-venv 激活和 Exa API key
- **解决方案**：Actions 更新仅依赖 GitHub 搜索，Exa 由 Hermes cron 处理
- **发现日期**：2026-07-06

### 7. 前端搜索无 debounce
- **现象**：618 条 × JSON.stringify 全文搜索，每次按键触发
- **原因**：input 事件无防抖
- **解决方案**：大数据量下可以考虑 debounce 或 Workers，当前 618 条可接受
- **发现日期**：2026-07-06

### 8. version bump 频繁
- **现象**：每次数据变化都 bump 版本号
- **原因**：update-data.yml 在每次 commit 前执行 bump_version.py
- **解决方案**：只在有意义的数据变更时 bump（当前无影响）
- **发现日期**：2026-07-06

### 9. common.py JSON/YAML 兼容性
- **现象**：`load_jsonish()` 只支持 JSON，但 `data/projects.yaml` 是 YAML 格式
- **原因**：原始代码假设所有数据文件都是 JSON，但 migrate_data.py 用 yaml.dump 写入
- **解决方案**：更新 `load_jsonish()` 先尝试 JSON 解析，失败则 fallback 到 YAML；`save_jsonish()` 按文件扩展名选择格式
- **发现日期**：2026-07-13

### 10. normalize.py rules 关键词误匹配
- **现象**：`resource_types_for('overrules nothing')` 错误匹配到 'rules' 类型
- **原因**：'rules' 作为子串出现在 'overrules' 中
- **解决方案**：将 'rules' 改为更精确的 'ruleset'、'rules file' 等组合词
- **发现日期**：2026-07-13

### 11. 数据迁移后质量门禁阈值需调整
- **现象**：迁移后 274 条记录（原 618 条），tool coverage 和 curated/rejected 阈值不再适用
- **原因**：原阈值针对 618 条数据设定（如 tool coverage >=10），274 条时 goose 只有 1 条
- **解决方案**：降低阈值为 tool coverage >=1, records >=100, curated >=20, rejected >=5
- **发现日期**：2026-07-13

### 12. build_site.py 多次运行产生双重 hash 文件名
- **现象**：build_site.py 连续运行两次后，index.html 中的 JS 引用变成 `js/i18n.9dd7a4.9dd7a4.js`（双重 hash），导致 404
- **原因**：第一次运行生成的 hash 文件（如 `i18n.9dd7a4.js`）被第二次运行的 `glob('*.js')` 当作源文件再次 hash
- **解决方案**：在生成新 hash 前执行三步清理：(1) 用正则将 index.html 中的 hash 引用恢复为原始文件名（支持多段 hash `(\.[a-f0-9]{6})+`）；(2) 删除旧的 hash 文件（`*.*.js` 和 `styles.*.css`）；(3) 只处理原始源文件（排除 hash 文件）
- **发现日期**：2026-07-13

### 14. 事件委托替代 inline onclick
- **现象**：第 2 批代码中动态生成的元素使用 inline onclick，导致 HTML 臃肿且难以维护
- **原因**：每个动态元素单独绑定事件，未使用事件委托
- **解决方案**：所有动态元素用 `data-action` + `data-id` 属性，在 document 上统一监听 click 事件，通过 `e.target.closest('[data-action]')` 判断点击类型
- **发现日期**：2026-07-13

### 15. JS 对象方法中箭头函数的 this 绑定问题
- **现象**：render.js 中 `pills: xs => (xs||[]).map(x => ...this.esc(x)...)` 报错，this 不指向 SIC_render
- **原因**：箭头函数不绑定自己的 this，继承外层作用域的 this（可能是 window/undefined）
- **解决方案**：对象方法改用普通函数（`pills: function(xs) {...}`），内部用 `SIC_render.esc(x)` 或 `var self = this` 模式
- **发现日期**：2026-07-13

### 16. gh repo view --json 不支持 readme 字段
- **现象**：`gh repo view <repo> --json readme` 报错 "Unknown JSON field: readme"
- **原因**：gh CLI 的 `--json` 字段列表不含 readme（有 nameWithOwner, stargazerCount, forkCount 等但不包括 readme）
- **解决方案**：分两步获取：(1) `gh repo view --json <标准字段>` 获取元数据；(2) `gh api repos/<repo>/readme --jq .content` 获取 README（base64 编码），然后 `base64.b64decode()` 解码
- **发现日期**：2026-07-13

### 17. licenseInfo.spdxId 为 None，需 fallback 到 key 字段
- **现象**：normalize.py 提取 license 时全部为 null，即使 repo-details.json 中有 licenseInfo
- **原因**：`gh repo view --json` 返回的 licenseInfo 中 `spdxId` 为 None，实际 license 信息在 `key` 字段（如 `mit`、`apache-2.0`）
- **解决方案**：先取 `spdxId`，为 None 时 fallback 到 `key` 字段；`NOASSERTION` 和 `none` 也视为 null
- **发现日期**：2026-07-13

### 18. GitHub 仓库重定向方向与预期相反
- **现象**：设计文档假设 `block/goose` 是正确路径、`aaif-goose/goose` 是旧路径，但实际相反
- **原因**：GitHub 仓库从 `block/goose` 重定向到 `aaif-goose/goose`（Block 收购后 org 迁移方向与预期不同），同理 `getcursor/cursor` -> `cursor/cursor`，`sst/opencode` -> `anomalyco/opencode`
- **解决方案**：用 `gh repo view <repo> --json nameWithOwner` 验证实际重定向方向，不要假设
- **发现日期**：2026-07-13

### 19. resource_type 误标根因：tutorial 规则过宽 + 无优先级
- **现象**：含 "Claude Code skill" 的项目被标为 tutorial 而非 skills
- **根因 1**：旧 `resource_types_for()` 遍历字典无优先级，tutorial 的 'guide' 关键词太宽泛
- **根因 2**：skills 规则用 'skill' 单词匹配但优先级与 tutorial 相同，两者同时匹配
- **解决方案**：(1) 分两阶段匹配：先检查具体类型（mcp/skills/extension/rules/framework/cli），再检查 tutorial；(2) tutorial 移除 'guide' 关键词；(3) skills 用更精确的短语如 'claude code skill'、'agent skill'
- **发现日期**：2026-07-13

### 20. 批次 B 前端审美重做的经验（v3）
- **现象**：从"数据库后台"风格重做为 Linear/Vercel 现代深色审美，同时修复 15 个 dogfood 交互问题
- **关键决策与坑**：
  1. **metrics.date 字段不存在**：build_site.py 生成的 metrics.json 没有 `date` 字段，footer 的"数据更新于"需要 fallback 到 `new Date().toISOString().slice(0,10)`。如需真实更新时间，需在 build_site.py 的 metrics 字典中添加 `'date': datetime.now().strftime('%Y-%m-%d')`
  2. **色彩标签 pill 类名**：resource_type 中的 `agent-framework` 对应 CSS 类名 `pill-type-agent-framework`（连字符保留），不能简化为 `pill-type-framework`
  3. **radiogroup 修复**：不能用 `toggleMode()` + `classList.toggle('active')`，因为点已激活的按钮也会翻转。正确做法：判断点击的是哪个按钮，如果新模式与当前相同则 return
  4. **writeState 保留 hash**：`history.replaceState(null, '', path + '?' + qs)` 会丢掉 hash。需在末尾追加 `+ location.hash`
  5. **Inter 字体需 Google Fonts 加载**：CSS 中声明 `font-family: 'Inter'` 但不在 HTML 中加载 Google Fonts 链接则不生效。用 `<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700;800&display=swap">`
  6. **build_site.py 的 hash 清理**：多文件同时修改后重新 build，旧的 hash 文件会因内容相同而保留（如 data.js/charts.js 未改则 hash 不变），这是正确行为，不是 bug
- **发现日期**：2026-07-13

### 22. 遗留项修复（seed-tools / resource_type / 字段填充）
- **现象**：三批次评估后仍有 seed-tools 路径错误、高星产品标 tutorial、字段填充率被低估、detail JSON 缺字段
- **关键坑与结论**：
  1. **GitHub 仓库迁移后的规范路径 vs 实际 nameWithOwner**：`gh repo view block/goose` / `getcursor/cursor` / `sst/opencode` 会成功，但返回的 `nameWithOwner` 可能是迁移后的 `aaif-goose/goose`、`cursor/cursor`、`anomalyco/opencode`。seed-tools 仍应写**规范/品牌路径**；projects 数据里可能继续保留实际 path，两者不必强行统一为同一字符串。
  2. **`forks=0` / `stars=0` 不是缺失**：用 `if not project.get('forks')` 会把合法 0 当成未填充，导致 enrich 反复请求且计划中的“truthy 填充率”被低估。正确判定：`key not in project or value is None`。
  3. **空 `topics: []` / `languages: []` 是已查询结果**：很多高星仓库（含 claude-code、opencode、cursor）在 GitHub 上本身无 topics。应持久化 `[]` 以便断点续传跳过，不能把 `[]` 当缺失。
  4. **`gh repo view --json` 不支持 `readme` 字段**：README 必须用 `gh api repos/{full}/readme --jq .content` + base64 解码（见 `collect_github.py` / `enrich_projects.py`）。
  5. **license 填充上限受 GitHub 本身限制**：大量长尾仓库 `licenseInfo=null`，再刷 API 也补不上；评估填充率时要区分“未查询”和“源站无 license”。
  6. **resource_type 误标会同时存在于 seed 官方项与同名 github 项**：修分类时要按 `repo` **和** `name` 双匹配，否则只修到一条副本。
  7. **deploy 需要显式 dest**：`deploy_site.py` 默认读 `SEARCH_IN_CODING_WEBROOT`，未设置时必须 `--dest /var/www/coding.lzpgood.online`。

  - seed-tools：goose=`block/goose`，cursor=`getcursor/cursor`，opencode=`sst/opencode`
  - 高星 tutorial 误标清零
  - present 填充：forks/stars/topics ≈ 99%；readme_preview ≈ 94%；license ≈ 60.5%（受源站限制）

- **发现日期**：2026-07-13

### 23. 第 3 批 LLM 分析系统的经验（2026-07-14）

- **现象**：实现每周 LLM 深度分析系统（SenseNova + DeepSeek-V4-Flash），遇到 API 调用、JSON 解析、key 轮询等问题
- **关键坑与结论**：

  1. **LLM JSON 输出不稳定**：LLM 经常用 markdown 代码块包裹 JSON，或在 JSON 前后加解释性文字。`parse_json_response()` 需要多级容错：直接解析 -> markdown 代码块提取 -> 正则匹配 -> 首尾花括号截取。最后一级（首尾花括号）处理嵌套 JSON 特别有效。

  2. **API key 轮询不是简单的 round-robin**：13 个 key 中可能有部分失效（401/403）。`KeyRotator` 需要跟踪失败 key 并跳过，全部失败时重置重试。注意 `mark_failed()` 后 `next()` 要从当前位置继续而非从头开始。

  3. **429 速率限制需要指数退避**：`time.sleep(5 * (attempt + 1))` 而非固定等待。3 个并发请求 + 13 key 轮询基本不会触发 429，但作为防御措施必须实现。

  4. **RateLimitError 类必须在 call_llm 引用前定义**：虽然 Python 运行时查找允许类定义在使用之后（只要在调用时已定义），但代码可读性要求异常类定义在使用之前。

  5. **不使用 delegate_task 调用 LLM**：cron 自动运行时无人审批，delegate_task 会被阻塞。直接用 urllib 调用 OpenAI 兼容 API 是正确方案。

  6. **Cron 时间冲突**：每日 cron 在 03:00 运行，每周 LLM 分析不能同时运行（LLM 分析可能 30-60 分钟）。解决方案：每周一 03:30 运行（每日 cron 通常几分钟完成）。

  7. **benchmark_selection_prompt 需要传 project id**：LLM 选择参照项目时需要知道项目 ID 才能回写。prompt 中必须包含 `id={p.get('id')}` 字段，否则 LLM 只返回项目名无法映射。

  8. **报告生成调用已有脚本而非内联 LLM**：`generate_reports.py` 已在前端重写 v2 时重写为基于数据的报告生成器。weekly_analysis.py 应调用它而非自己用 LLM 生成报告（避免重复工作和 LLM 输出不稳定）。

  9. **merge_analysis_result 需要 deepcopy**：直接修改传入的 project dict 会影响原始数据（特别是 prev_projects 快照）。使用 `copy.deepcopy()` 确保安全。

  10. **snapshot 文件不入 Git**：`data/snapshots/` 目录已在 .gitignore 中排除，避免每周快照膨胀 Git 仓库。

- **发现日期**：2026-07-14

### 25. 第 3 批 P0/P1 修复（cron 超时 / score_detail 覆盖 / 无 checkpoint / 前端 /60）
- **现象：**
  1. Hermes cron 默认脚本超时 120s，weekly 全量 16–48 分钟必被杀；daily job 也已出现 `Script timed out after 120s`
  2. `merge_analysis_result` 把 LLM `quality_detail` 写入 `score_detail`，覆盖可量化 stars/activity，详情页 Stars/Activity 显示 0
  3. 分析完才统一保存，中断后整轮白跑
  4. 已分析 total=87–92 仍展示 `/60`，进度条溢出
  5. `benchmark_ref` 直接显示 project_id（如 `official-hermes-agent`）
- **解决方案：**
  1. `hermes config set cron.script_timeout_seconds 3600`
  2. `quality_detail` 独立字段；`score.py` 只写 quantifiable `score_detail`
  3. `run_analysis` 每批 3 个后 checkpoint 保存 projects.yaml
  4. `quality_score > 0` 展示 `/100`，否则 `/60`
  5. 前端用 project id 查 name 展示
  6. official-seed 强制 `tracking_priority=track`（Cursor 被 LLM 标 index 导致 quality_gate FAIL）
- **发现日期：** 2026-07-14


## 教训清单

1. **分类规则要尽量精确**：太宽泛的分类导致后续评分和推荐偏斜。关键词匹配要有优先级和排除逻辑。
2. **来源标注必须透明**：fallback 数据就是 fallback，绝不能标记为 Exa 数据。用户需要知道数据可信度。
3. **GitHub 优先原则**：GitHub 数据质量最高，评分优先，curated 优先。Exa 数据补充，fallback 兜底。
4. **架构简单优先**：纯文件存储 + 静态站点的方案避免了数据库运维和前后端 API 调试的复杂度。
5. **可复用设计比一次性方案好**：替换配置即可迁移到其他生态追踪，增加了项目价值。
6. **安全响应头要在上线前配置**：HSTS 和 CSP 是公开站点的必需品，不能事后补。
7. **testing-review-ci 分类太宽**：PR review、CI automation、code review 关键词匹配太多 GitHub 项目。
8. **前端交互必须逐个测试**：第 2 批重写产出了代码但 10 个 bug 导致功能不可用。每个交互路径（发现区、报告链接、工具卡片、虚拟滚动、筛选、图表、详情面板、收藏、markdown 渲染、移动端）都必须实际验证，不能假设"代码写了就等于功能可用"。
9. **事件委托优于 inline onclick**：动态生成的元素用 data-action + 事件委托比 inline onclick 更干净、更易维护，且避免 HTML 中混入 JS 代码。
10. **JS 对象方法慎用箭头函数**：对象字面量中的方法如果用箭头函数，this 不绑定到对象本身。改用普通函数声明或直接引用对象名。
11. **gh CLI 字段需实测验证**：`gh repo view --json` 支持的字段列表与 GitHub API 不同（如不支持 readme），不要假设 API 文档中的字段都可用。用 `gh repo view <repo> --json invalidfield` 查看可用字段列表。
12. **GitHub 重定向方向不可假设**：仓库迁移后的重定向方向可能与预期相反。用 `gh repo view --json nameWithOwner` 实测验证实际路径。
13. **License 字段需要 fallback 策略**：gh CLI 返回的 licenseInfo 中 spdxId 经常为 None，需要 fallback 到 key 字段。
14. **分类规则需要优先级机制**：多个分类规则可能同时匹配，需要明确的优先级顺序（具体类型 > 通用类型），否则会产生误标。
15. **前端审美重做需同步更新 wiki**：L4A 的 CSS 变量表、模块方法表、交互模式描述在审美重做后必须同步更新，否则 wiki 与代码脱节，新 Agent 读 wiki 会得到错误信息。
16. **metrics.json 无 date 字段**：build_site.py 生成的 metrics 不含更新日期，如需在页脚显示真实更新时间需在 build_site.py 中补充。

17. **翻译质量门禁不能只看“有结果”**：repo 名/产品名场景下 LLM 常原样返回英文；必须强制“含中文 + 不等于原文”才写回缓存/yaml。
18. **批量翻译要可断点续传**：URL hash 缓存 + 可重复运行，比一次性全量失败重来更稳。
19. **字段填充率要区分 present vs truthy**：`forks=0`、`topics=[]` 是合法填充；用 truthy 统计会把真实数据误判为缺失。
20. **enrich 断点续传必须持久化空结果**：否则会对无 topics/无 license 的仓库无限重试。

21. **LLM JSON 输出需要多级容错解析**：LLM 经常用 markdown 代码块包裹 JSON 或在前后加文字。解析器需要：直接解析 -> 代码块提取 -> 正则匹配 -> 首尾花括号截取四级 fallback。

22. **Cron LLM 分析不能用 delegate_task**：cron 自动运行时无人审批，delegate_task 会被阻塞。必须用 urllib 直接调用 OpenAI 兼容 API。同时注意与每日 cron 的时间冲突（错开 30 分钟）。

23. **API key 轮询要跟踪失败状态**：多 key 轮询不是简单 round-robin，需要 `KeyRotator` 跟踪失败 key 并跳过，全部失败时重置。benchmark prompt 必须传 project id 才能回写。

24. **报告生成复用已有脚本**：`generate_reports.py` 已存在且基于数据生成报告，weekly_analysis.py 应调用它而非自己用 LLM 生成报告，避免重复工作和 LLM 输出不稳定。

25. **可量化分项与 LLM 分项必须分字段**：`score_detail` 只放 stars/activity/adoption/maturity；LLM 的 relevance 等必须写 `quality_detail`，否则前端会把 Stars 读成 0。

26. **长任务 cron 必须显式提高 script_timeout**：Hermes 默认 120s；周分析 16–48 分钟要设 `cron.script_timeout_seconds=3600`。

27. **批处理要增量落盘**：每批（不是每条）保存 checkpoint，超时/中断后可续跑。

28. **official-seed 的 tracking_priority 不能完全交给 LLM**：质量门禁要求 official 一律 track；LLM 可能把低质量官方页标成 index/reject，需代码强制覆盖。
